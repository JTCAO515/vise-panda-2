# VisePanda v5.0.9 生产稳定性专项设计

## 概述

本轮目标不是新增功能，而是把线上正式站点修到“稳定、可响应、可点击、可浏览”的状态。  
当前用户在真实使用中已经遇到以下问题：

- 首页与多个 tab 加载慢、响应度低
- 图片无法加载
- `Sign in` 点击无响应
- 手机端 tab / 导航不可见

因此，本轮定义为：`v5.0.9 Production Stability Pass`。  
优先级高于功能迭代、高于视觉继续打磨。

## 目标

本轮完成后，线上正式站点需要满足以下目标：

1. 主导航与主要 tab 在桌面端和手机端都稳定可见、可点击
2. `Sign in` 入口能稳定触发正确流程，不再表现为“点击无响应”
3. 首页与主要视图的关键图片能正确显示；失败时页面也不会塌陷
4. 页面切换与关键按钮具备明确反馈，不再让用户感到“点了没反应”
5. 前端初始化链路更稳，不因为单点异常导致多个页面一起失效
6. 为这些问题补上可回归的结构/行为测试

## 非目标

本轮明确不做以下内容：

- 新功能扩展
- 新的城市/行程/工具能力
- 全站视觉重做
- 后端架构升级
- 无关模块的大规模重构

只允许做与“生产稳定性”直接相关的小范围结构收口。

## 问题拆解

### 1. 启动与首屏问题

已知表现是加载慢、响应低，说明首屏可能存在以下风险之一或多个叠加：

- 初始化逻辑过重，首屏一次性执行太多工作
- 首屏渲染依赖多个资源或接口，缺少渐进式加载
- 某个模块初始化失败后阻断后续交互绑定

### 2. 交互链路问题

`Sign in` 点击无响应通常意味着：

- 事件未绑定
- 事件在初始化异常后没有执行
- 元素被 overlay / layout 遮挡
- 依赖的弹层或状态容器未正确挂载

多个 tab 都有问题，则更可能是“公共初始化层”或“导航/视图切换层”存在稳定性缺陷。

### 3. 资源加载问题

图片不显示通常意味着以下几类问题：

- 静态资源路径不兼容线上部署环境
- 图片 URL 拼接错误
- 图片目录真实不存在或命名不一致
- 资源失败后没有 fallback，页面直接显示破图或空白

### 4. 移动端布局问题

手机端 tab 不可见通常与以下因素相关：

- header / bottom nav / safe-area 冲突
- `100vh` 与移动端视口实际高度不一致
- `overflow: hidden` 误伤内容层
- z-index 或媒体查询意外隐藏导航

## 设计方案

### 方案总览

本轮采用“生产稳定性专项重整”的方式推进，但实现上拆成 5 个明确工作面，按先止血后收口的顺序完成：

1. 线上行为复现与证据采集
2. 交互链路修复
3. 资源与图片链路修复
4. 页面响应与加载反馈优化
5. 移动端导航与布局稳定性修复

### 工作面一：线上行为复现与证据采集

先在真实线上正式站点复现问题，并采集以下证据：

- Console 报错
- Network 中失败请求
- 静态资源 404 / 路径错误
- DOM 层级、遮挡、不可点击区域
- 手机端视口下导航是否被隐藏或推出可视区域

输出不是文档，而是明确的问题清单和根因分类，用于决定修复顺序。

### 工作面二：交互链路修复

优先修复公共入口层，尤其是：

- `Sign in`
- 顶部导航 / tab
- 主 CTA
- modal 打开/关闭

设计要求：

- 入口按钮必须在 DOM、事件绑定、可点击层级三个层面同时成立
- 初始化失败不能让整个站点变成“点击无响应”
- 对关键入口增加最小反馈：
  - `loading`
  - `disabled`
  - `toast/error`

如现有 `web/app.js` 中公共初始化过于集中，允许做与稳定性直接相关的小范围拆分，例如：

- `bootstrap`
- `bindGlobalUI`
- `bindAuthTriggers`
- `bindNav`
- `bindModalClose`

但不把这一轮演变成全面模块化重构。

### 工作面三：资源与图片链路修复

目标是让线上图片“能显示”，并在失败时“页面不难看”。

设计要求：

- 检查并统一图片路径来源，避免本地可用、线上失效
- 给关键图片增加失败 fallback
- 首屏关键图片优先，非关键图片允许延迟加载
- 即使图片失败，也保留稳定占位，避免布局塌陷

必要时会补充：

- 占位容器
- fallback class
- `onerror` 或统一图片错误处理
- 对首屏图片做更保守的加载策略

### 工作面四：页面响应与加载反馈优化

这部分目标不是极致性能优化，而是修复“主观卡死感”。

设计要求：

- 首页与主要视图先渲染壳层，再加载次级内容
- tab 切换时优先展示容器与反馈，而不是静默等待
- 对可能较慢的模块提供：
  - skeleton
  - loading text
  - retry / error message

如果某些页面在进入时会同步执行不必要逻辑，则调整为：

- 首屏最小初始化
- 按视图惰性加载
- 错误隔离，不让单页失败拖垮全局

### 工作面五：移动端导航与布局稳定性修复

手机端重点不是“更漂亮”，而是“导航一定能看到、一定能点到”。

设计要求：

- 检查并修复 tab/nav 在竖屏下的可见性
- 保证 header、bottom nav、内容层、safe-area 之间不互相覆盖
- 避免媒体查询把关键导航隐藏
- 处理 `100vh` / `100dvh` / `env(safe-area-inset-bottom)` 的组合

这部分会优先修复结构问题，再做少量样式补丁，不重做整体设计语言。

## 预期改动范围

预计会主要落在以下文件或同类文件：

- `web/index.html`
- `web/app.js`
- `web/app.css`
- `web/admin.html`（如主入口或 auth 触发与 admin 共享）
- `web/tests/*.test.js`
- `README.md`
- `HANDOFF.md`
- `CHANGELOG.md`

如问题与静态资源路径有关，可能涉及：

- `static/`
- 图片目录引用
- 资源 URL 生成逻辑

原则上不改后端业务逻辑；只有在 `config` 或静态资源分发与线上资源问题直接相关时，才允许极小范围修补。

## 错误处理设计

### 初始化错误

- 公共初始化异常不应阻断全站基础交互
- 对关键初始化步骤使用分段保护
- 单个子模块失败时，页面仍应保留主导航和基本视图切换能力

### 资源错误

- 图片失败时切换到 fallback 状态
- 不允许出现大块空白或布局塌陷

### 交互错误

- 登录/弹层/切换失败时必须给出用户可感知反馈
- 禁止“点击了但完全没反应”

## 测试设计

### 前端结构/行为测试

本轮至少补齐以下测试覆盖：

1. `Sign in` 入口存在且可触发
2. 主导航 / tab 在桌面与移动端结构下都存在
3. 图片 fallback 容器或错误状态结构存在
4. loading / error / skeleton 容器存在
5. Tools / Cities / Trips / Chat 在切换时的基础挂点不缺失

### Smoke Test

本地手动回归至少覆盖：

1. 首页加载
2. `Sign in`
3. Chat
4. Cities
5. Trips
6. Tools
7. 手机端视口下导航与 tab

### 线上回归基线

以上修复最终以线上正式站点结果为准，验收基线就是用户已报告的 5 类问题：

1. 加载慢
2. 响应低
3. 图片无法加载
4. `Sign in` 无响应
5. 手机端 tab 不可见

## 执行顺序

按以下顺序推进：

1. 复现线上问题并记录根因
2. 修 `Sign in` 与主导航/tab 可用性
3. 修图片与静态资源链路
4. 修首屏与 tab 切换响应
5. 修手机端 tab/nav 可见性与布局
6. 补测试、同步版本、更新文档、发布

## 风险与边界

### 风险

1. 可能存在多个根因叠加，而不是单点 bug
2. 线上问题可能与本地行为不完全一致
3. 公共初始化层修复可能影响多个页面

### 控制方式

- 先采证据，再动修复
- 以线上行为为第一依据
- 每修一类问题，都对主 tab 做回归
- 不借机做无关重构

## 验收标准

本轮通过的标志是：

- 线上主站关键 tab 都能稳定进入
- `Sign in` 能稳定触发正确流程
- 关键图片能显示，失败时有可接受兜底
- 页面切换和关键按钮具备明确反馈
- 手机端竖屏下能稳定看到并点击主要 tab/nav
- 前端测试和本地 smoke 不出现明显回归
