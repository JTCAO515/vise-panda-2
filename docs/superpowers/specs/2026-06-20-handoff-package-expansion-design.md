# VisePanda 交接包扩展设计

## 概述

当前仓库已经具备：

- `HANDOFF.md` 作为主交接入口
- `docs/2026-06-20-engineering-handoff-notes.md` 作为详尽工程附录
- `docs/2026-06-20-commercial-upgrade-plan.md` 作为商用升级路线文档

但如果目标是把项目正式交给其他人持续接手，仅靠“主交接文档 + 工程附录”还不够。  
新接手者通常还会缺 3 类直接可执行的材料：

1. 第一周具体该怎么进入项目
2. 哪些高风险文件不要随便动
3. 线上发布前后如何做回归检查

因此，本轮要补齐一套更完整的交接包，而不是继续把内容堆进 `HANDOFF.md`。

## 目标

本轮完成后，仓库应具备一套可直接交付给新接手者的文档包，覆盖：

1. 项目总览与交接入口
2. 工程全景与主链路说明
3. 新接手开发者的首周进入方式
4. 高风险文件与高风险修改点说明
5. 线上回归与发布前后检查手册

目标不是写成“漂亮文档”，而是让新接手者：

- 能快速理解项目
- 不容易误改高风险部分
- 能按正确顺序进入工作
- 能知道如何做上线前后回归

## 非目标

本轮不做以下内容：

- 不修改产品功能
- 不修改测试代码
- 不更新版本号
- 不改变部署方案
- 不新增新的技术实现

本轮是**文档层补全与结构化交接增强**。

## 方案选择

### 方案 A：继续扩写 `HANDOFF.md`

优点：

- 所有信息都在一个文件里
- 无需新增文档

缺点：

- `HANDOFF.md` 会继续膨胀
- 新接手者不容易快速定位需要的内容
- 维护成本会越来越高

### 方案 B：继续只保留主文档 + 工程附录

优点：

- 文件数量少
- 当前结构已经初步具备

缺点：

- 仍缺“怎么接手”“哪些地方危险”“怎么回归”这类执行型材料
- 交接依然更像“说明”，不像“工作包”

### 方案 C：补齐完整交接包

本轮采用方案 C。

在现有基础上新增 3 份文档：

1. `docs/2026-06-20-first-week-takeover-checklist.md`
2. `docs/2026-06-20-high-risk-files-guide.md`
3. `docs/2026-06-20-production-regression-manual.md`

并将这些文档接入：

- `HANDOFF.md`
- `README.md`

优点：

- 文档职责清晰
- 最适合真正的项目交接
- 新人、开发者、测试/产品都能找到适合自己的入口

缺点：

- 文件数量增加
- 索引关系必须写清楚

## 设计方案

## 1. 首周接手清单

新增文件：

`docs/2026-06-20-first-week-takeover-checklist.md`

### 目标

这份文档是给新接手开发者使用的，不是长期 roadmap。  
它的作用是防止新人在第一周就：

- 一上来重构
- 一上来改部署
- 一上来误删兼容层
- 一上来跳过测试直接改核心文件

### 结构

文档至少覆盖以下章节：

1. 文档目的
2. 接手前准备
3. 第 1 天
4. 第 2-3 天
5. 第 4-5 天
6. 这一周不要做什么

### 内容原则

- 强调理解顺序，而不是机械任务列表
- 解释“为什么先做这个”
- 强调先看、先跑、先验证，再改代码

## 2. 高风险文件修改指南

新增文件：

`docs/2026-06-20-high-risk-files-guide.md`

### 目标

这份文档告诉接手者：

- 哪些文件风险最高
- 为什么高风险
- 改之前要先确认什么
- 改完至少测什么

### 重点覆盖文件

至少包括：

- `web/app.js`
- `web/app.css`
- `web/index.html`
- `api/auth.py`
- `api/chat.py`
- `api/index.py`

### 每个文件的固定说明结构

每个高风险文件都采用统一格式：

1. 这个文件负责什么
2. 为什么危险
3. 修改前先确认什么
4. 修改后至少做什么检查

### 补充章节

增加一节“相对安全的改动区”，告诉接手者哪些地方更适合先做小修改，比如：

- 纯文案
- 某些独立数据文件
- 小范围样式
- 低耦合展示层

## 3. 线上回归检查手册

新增文件：

`docs/2026-06-20-production-regression-manual.md`

### 目标

这份文档不是写给自动化测试框架，而是写给：

- 接手开发者
- 产品
- 测试
- 发布前后做人工回归的人

### 结构

至少包括：

1. 文档用途
2. 回归前提
3. 桌面端核心回归
4. 移动端核心回归
5. 资源与性能观察点
6. 如果发现问题先怀疑哪里
7. 发布前最低通过线

### 核心覆盖范围

桌面端至少覆盖：

- 首页
- Sign in
- Chat
- Cities
- Trips
- Tools

移动端至少覆盖：

- bottom nav
- tab 可见性
- safe-area
- 页面是否被遮挡
- 输入区是否被挤压

### 内容原则

- 不是机械 checklist，而是“带判断说明的手册”
- 要写清楚发现问题后先看哪里

## 4. 主文档与索引联动

本轮不会重新大改 `HANDOFF.md` 的主体结构，但会在其中补充这 3 类文档入口：

- `first-week-takeover-checklist`
- `high-risk-files-guide`
- `production-regression-manual`

同时更新 `README.md` 的 `Planning Docs`，加入这些新文档链接。

### 主文档中需要体现的变化

1. 在“接手建议”里引用首周清单与高风险文件指南
2. 在“关键文档索引”里加入 3 个新文档

### README 中需要体现的变化

在 `Planning Docs` 中增加：

- `First-Week Takeover Checklist`
- `High-Risk Files Guide`
- `Production Regression Manual`

## 错误控制与边界

为避免交接包继续失控膨胀，本轮遵循以下约束：

1. 每份文档只承担一种明确职责
2. 不在新文档里大段重复 `HANDOFF.md`
3. 不在主文档中重新粘贴完整工程细节
4. 索引必须明确，不能让读者自己去仓库盲找

## 验收标准

本轮通过的标准是：

1. 仓库中存在 3 份新增交接附录
2. 每份附录都能独立阅读
3. `HANDOFF.md` 中能找到这些文档入口
4. `README.md` 中也能找到这些文档入口
5. 一个新接手的人不需要再依赖聊天记录，也能知道：
   - 第一周怎么接
   - 哪些文件危险
   - 上线前后怎么验收

## 交付方式

这轮不涉及产品版本升级。  
只会：

- 新增交接附录文档
- 更新 `HANDOFF.md`
- 更新 `README.md`
- 提交并推送到 GitHub
